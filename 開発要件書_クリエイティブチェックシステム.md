> 作成日: 2026-03-08

---

## 1. プロジェクト概要

### 目的
広告クリエイティブ（バナー・動画・TD）のレギュレーションチェックを属人化から仕組み化するWebシステムを構築する。

### 背景
- レギュレーションチェックが個人の知識・記憶に依存しており、NG表現の見落としによる事故が繰り返し発生
- レギュレーション変更頻度が高く（特にアール社案件）、最新ルールの把握が困難
- CR品質向上WGの方針：「仕組みとツールで防ぐ体制」への転換

### スコープ
- **対象CR**: バナー（静止画）、動画、タイトル（TD）
- **対象外**: 記事LP（別フローで管理のため）

## 2. システム構成

### アーキテクチャ決定事項

| 項目 | 決定内容 | 理由 |
|------|---------|------|
| **構成** | ad-dashboardとは独立したアプリ | ad-dashboardがLightsailメモリ制限で既に逼迫しているため。障害隔離・安定性重視 |
| **技術スタック** | React + Express + PostgreSQL + Prisma + TypeScript | ad-dashboardと同じ構成で社内の技術統一 |
| **CSS** | Tailwind CSS | 同上 |
| **ビルド** | Vite | 同上 |
| **認証** | JWT（独立したユーザー管理） | ad-dashboardとは別管理 |
| **インフラ** | AWS Lightsail（別インスタンス） | 軽量アプリのため最小プランで十分 |

## 3. コア機能: 案件選択 → チェック フロー

### 3-1. 操作フロー

```
① 案件選択（クライアント × 商材）
   例: ファーマフーズ × ニューモ育毛剤
        ↓
② 媒体選択（Meta / Google / ByteDance / LINE / SmartNews）
        ↓
③ CR種別選択（バナー / 動画 / TD）
        ↓
   ※ ①②③の選択に基づき、適用ルールが自動絞り込み:
     - 全社共通ルール（法令系）
     - 該当クライアント固有ルール
     - 該当媒体固有ルール
     - 該当商材固有ルール
        ↓
④ テキスト入力 → NG表現自動チェック
⑤ チェックリスト表示 → 手動確認（OK/NG/未確認）
⑥ 二重チェック者の記名
        ↓
⑦ 全項目OK → 提出許可
   NG項目あり → ブロック＋理由表示
        ↓
⑧ チェックバック結果登録（CL判定: ○/×/△）
   × → NG事例としてDBに自動蓄積
```

### 3-2. ポイント
- **案件選択が起点**: 案件ごとにレギュレーションが異なるため、事前選択で適用ルールを絞り込むことでチェック精度を上げる
- **過去実績の活用**: CR提出シートの過去○×△データを参照し、類似NG表現を警告する

## 4. データベース設計

### 4-1. テーブル一覧

```prisma
// ===== マスタ系 =====

model Client {
  id        Int       @id @default(autoincrement())
  name      String    @unique  // "ファーマフーズ", "明治薬品", "フューチャーラボ"
  products  Product[]
  rules     RegulationRule[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
}

model Product {
  id        Int       @id @default(autoincrement())
  name      String    // "ニューモ育毛剤", "ラクトロン", "シボラナイト"
  clientId  Int
  client    Client    @relation(fields: [clientId], references: [id])
  rules     RegulationRule[]
  materials MaterialLicense[]
  submissions CrSubmission[]
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt

  @@unique([clientId, name])
}

// ===== レギュレーション系 =====

model RegulationRule {
  id              Int       @id @default(autoincrement())
  category        String    // "法令" | "クライアント" | "媒体" | "社内"
  subcategory     String    // "ステマ規制", "薬機法", "ファーマフーズ共通", "Meta"
  ruleText        String    // ルール内容（人が読む説明文）
  ngExpressions   Json?     // NG表現リスト ["愛用者", "購入者", ...]
  okExpressions   Json?     // OK代替表現 ["モニター", "体験者", ...]
  applicableMedia Json?     // 適用媒体 ["Meta", "Google", ...] null=全媒体
  clientId        Int?      // null=全クライアント共通
  client          Client?   @relation(fields: [clientId], references: [id])
  productId       Int?      // null=クライアント内全商材共通
  product         Product?  @relation(fields: [productId], references: [id])
  effectiveFrom   DateTime
  effectiveUntil  DateTime? // null=現在有効
  severity        String    // "絶対NG" | "要確認" | "推奨"
  sourceUrl       String?   // 根拠となるスプシ・ドキュメントURL
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt
}

model NgCase {
  id            Int       @id @default(autoincrement())
  clientId      Int
  productId     Int
  crType        String    // "バナー" | "動画" | "TD"
  ngExpression  String    // NG判定された表現
  ngReason      String?   // NG理由（CLフィードバック等）
  occurredAt    DateTime
  severity      String    // "事後NG" | "事前NG" | "ヒヤリハット"
  countermeasure String?  // 対策内容
  sourceSheetId String?   // 元のCR提出シートID
  createdAt     DateTime  @default(now())
}

// ===== チェック系 =====

model CheckTemplate {
  id         Int    @id @default(autoincrement())
  crType     String // "バナー" | "動画" | "TD" | "入稿"
  clientId   Int?   // null=共通テンプレート
  checkItems Json   // [{item: "NG表現セルフチェック", required: true}, ...]
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt
}

model CheckSession {
  id            Int       @id @default(autoincrement())
  productId     Int
  media         String    // "Meta" | "Google" | "ByteDance" | "LINE" | "SmartNews"
  crType        String    // "バナー" | "動画" | "TD"
  inputText     String?   // チェック対象のテキスト
  checkResults  Json      // [{itemId: 1, status: "OK"}, {itemId: 2, status: "NG", note: "..."}]
  ngMatches     Json?     // 自動検出されたNG表現 [{expression: "愛用者", rule: "ステマ規制", severity: "絶対NG"}]
  checkerUserId Int       // セルフチェック者
  reviewerUserId Int?     // 二重チェック者
  status        String    // "チェック中" | "二重チェック待ち" | "提出可" | "NG差戻し"
  submittedAt   DateTime?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
}

// ===== CR提出シート同期系 =====

model CrSubmissionSheet {
  id             Int       @id @default(autoincrement())
  productId      Int
  spreadsheetId  String    // Google Sheets ID
  sheetName      String?   // シート名（デフォルトは1枚目）
  lastSyncAt     DateTime?
  totalRows      Int       @default(0)
  createdAt      DateTime  @default(now())
  updatedAt      DateTime  @updatedAt

  @@unique([spreadsheetId, sheetName])
}

model CrSubmission {
  id              Int       @id @default(autoincrement())
  productId       Int
  product         Product   @relation(fields: [productId], references: [id])
  rowNumber       Int       // シート内の行番号
  submissionDate  DateTime? // 記載日
  text            String?   // テキスト内容
  charCount       Int?      // 文字数
  selfCheck       String?   // "○" | "△" | "×"
  firstClCheck    String?   // 1次CL確認 "○" | "×"
  firstClDate     DateTime?
  firstClNote     String?
  secondClCheck   String?   // 2次CL確認 "○" | "×"
  secondClDate    DateTime?
  secondClNote    String?
  finalResult     String?   // 最終可否
  finalNote       String?   // 最終可否備考（NG理由等）
  sheetId         String    // 元のスプレッドシートID
  createdAt       DateTime  @default(now())
  updatedAt       DateTime  @updatedAt

  @@unique([sheetId, rowNumber])
}

// ===== 素材管理系 =====

model MaterialLicense {
  id                Int       @id @default(autoincrement())
  materialName      String    // 素材名
  personName        String?   // 出演者名（例: "及川洋子"）
  applicableProducts Product[]
  licenseUntil      DateTime?
  status            String    // "有効" | "期限切れ" | "使用終了"
  createdAt         DateTime  @default(now())
  updatedAt         DateTime  @updatedAt
}

// ===== ユーザー系 =====

model User {
  id                 Int      @id @default(autoincrement())
  email              String   @unique
  name               String
  passwordHash       String
  role               String   @default("member") // "admin" | "member"
  mustChangePassword Boolean  @default(true)
  createdAt          DateTime @default(now())
}
```

## 5. 機能要件（Phase別）

### Phase 1: チェックリスト + NG表現DB（P0 - MVP）

#### 1-A. NG表現データベース構築
- 既存Googleスプレッドシートからの初期インポート機能
  - ファーマフーズG使用禁止表現シート
  - ファーマG・ニコリオ各案件NG一覧シート
- NG表現のCRUD管理画面（管理者用）
- カテゴリ別・クライアント別・媒体別のフィルタリング
- NG表現とOK代替表現のペア管理

#### 1-B. チェックリスト機能
- 案件（クライアント×商材）→ 媒体 → CR種別 の3ステップ選択UI
- 選択に応じた該当チェック項目の自動表示
- 各項目にOK/NG/未確認を記録
- NG項目があれば提出ブロック（警告表示）
- 二重チェック者の記名・承認フロー

#### 1-C. CR提出シート同期
- Google Sheets APIによる案件別CR提出シートの日次自動同期
- 新規行の差分取得（前回同期以降の行を取得）
- ○×△判定の更新検知
- ×判定が新たについた表現をNG事例（NgCase）に自動登録

### Phase 2: テキスト自動チェック（P1）

- テキスト入力欄にコピーを貼り付けると、NG表現DBと即時照合
- マッチしたNG表現をハイライト表示（赤: 絶対NG、黄: 要確認）
- 代替OK表現をインライン提案
- 過去のCR提出シートから類似NG表現を検索・警告
  - 例: 「根こそぎ」を入力 → 過去に「根こそぎNG」とフィードバックされた事例を表示

### Phase 3: 素材使用期限アラート（P1）

- 素材・出演者の使用期限管理
- 期限30日前・7日前・当日のアラート通知
- 対象商材との紐付け表示

### Phase 4: 外部連携強化（P2）

- **Google Sheets API**: レギュレーションシートの定期同期（NG表現ルール自動更新）
- **Chatwork API**: レギュレーションルーム（ID: 335308872）の新着メッセージを監視し、レギュ更新を検知・通知

### Phase 5: ダッシュボード（P3）

- レギュレーション更新履歴の一覧表示
- NG発生件数の案件別・CR種別集計
- ヒヤリハット共有の集約表示
- 月次レビュー用レポート

## 6. チェック対象CR別の必須チェック項目

### 6-1. バナー（静止画）
- [ ] NG表現セルフチェック（△ or ×記載）
- [ ] 未チェックフォルダに格納済みか
- [ ] 提出シートに転記済みか
- [ ] 二重チェック完了（最終：案件ディレクター）
- [ ] 完全NG → 削除、微妙 → 提出シートに△/×記載して提出

### 6-2. 動画
- [ ] 動画内テキスト起こしと実際の音声が一致しているか
- [ ] ナレーター文言・注釈も提出シートに記載されているか
- [ ] 体験者コメントに「依頼に基づく内容」表記があるか（ステマ対策）
- [ ] 動画がゴミ箱に入っていないか
- [ ] 修正時：変更部分のテキストを赤色に変更

### 6-3. タイトル（TD）
- [ ] タイトル集シート内で作成されているか
- [ ] NG表現セルフチェック（△ or ×記載）
- [ ] 二重チェック完了（最終：案件ディレクター）

### 6-4. 入稿時チェック（全CR共通）
- [ ] 記事URLと遷移先LPの整合性確認
- [ ] AD Ebisの広告ID備考欄に当該LPの内容を明記
- [ ] ダブルチェック者の記名

## 7. レギュレーション体系（DBに組み込むルール）

### 7-1. 全社共通（法令系）

| ルール | 詳細 | 重要度 |
|--------|------|--------|
| ステマ規制 | 口コミ表現: NG=「愛用者」「購入者」「リピーター」「投稿者」「夢中」「熱中」 / OK=「モニター」「体験者」「利用者」「使用者」 | 絶対NG |
| 薬機法 | 用法用量の指定NG（「食前に」→「食前などの」、「1回4粒」→「1回4粒目安」）、効果効能の断定不可 | 絶対NG |
| 景品表示法 | 虚偽の受注速報・閲覧人数・アンケート結果 → 一切禁止 | 絶対NG |
| 行政固有名詞 | 「消費者庁」等 → 全案件禁止 | 絶対NG |

### 7-2. クライアント別

| クライアント | 主要ルール |
|-------------|-----------|
| ファーマフーズG | NHK文言自体がNG、正露丸画像NG、「今だけ」→「今なら」、定期商材認識文言必須、リアル内臓・便系NG |
| 明治薬品 | 事前チェック必須、シボラナイト×工場コンテンツ併用NG、退職社員画像NG |
| フューチャーラボ | チェックバック前に配信可（NGなら即停止） |
| キュラ | 「入れ歯やめた」等の改善暗示NG、卵アレルギー注意記載 |

### 7-3. 媒体別

| 媒体 | 主要ルール |
|------|-----------|
| Meta | お腹の脂肪系文言+お腹素材の同時使用NG、ビフォーアフター風要対策、アカウント名にも効果効能避ける |
| Google | 医薬品審査フォーム提出必要 |
| ByteDance | 審査精度高い、他媒体OKでもBDではNGの表現あり |
| LINE | 審査否認理由の問い合わせフローあり |
| SmartNews | 独自審査基準あり |

## 9. 過去の重大事故（システムで防ぐべき事例）

| # | 事象 | このシステムでの防止策 |
|---|------|---------------------|
| 1 | ファーマNG表現（NHK、「脂肪と便まとめて出す」）が配信された | → テキスト自動チェックでNHK・過去NG表現を検出しブロック |
| 2 | バナーは商品Aだが遷移先LPが商品B | → 入稿時チェックリストで整合性確認を必須化 |
| 3 | Meta広告アカウント名がレギュ違反 | → 媒体別ルールのチェックリストに含める |
| 4 | 動画テキスト起こしミス、提出URL/番号ミス | → 動画チェックリストで確認を必須化 |
| 5 | ステマ対策のPR表記漏れ | → 全社共通ルールとして自動チェック |
