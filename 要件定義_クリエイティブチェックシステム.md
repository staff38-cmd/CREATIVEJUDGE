# クリエイティブチェックシステム 要件定義書

> 作成日: 2026-03-08
> 目的: 広告品質管理業務の属人化を解消し、誰でもレギュレーションに準拠したチェックができるシステムを構築する

---

## 1. 背景・課題

### 現状の問題
- レギュレーションチェックが**属人的な情報管理**に依存している
- NG表現の認識不足によるミスが繰り返し発生（貴秘薬NHK混入事故、meta広告アカウント名違反等）
- レギュレーション変更頻度が高く（特にアール社案件）、3ヶ月前の基準が現在ではNGになる可能性がある
- CR品質向上WGの結論:「気合・根性で乗り切る文化から、**仕組みとツールで防ぐ体制**への転換が必要」

### 現行チェックフロー
1. CR制作者がセルフチェック
2. 案件ディレクターが二重チェック（バナー/動画/TD）、記事は田中（章）
3. アール社に提出（提出シートに転記）
4. クライアントチェック → OK/NG判定
5. 事後NGが出たら、全記事横断で修正 → 再チェック → 再提出

## 2. レギュレーション体系（システムに組み込むべきルール）

### 2-A. 全社共通レギュレーション（法令系）

#### ステマ規制（景品表示法、令和5年10月施行）
- 記事冒頭に「広告：〇〇」「PR：〇〇」と記載必須
- フォントサイズ**11px以上**（Squad Beyondの最小文字10pxではNG）
- 画像内ではなくテキストで対応すること
- 口コミ表現ルール:
  - **NG**: 「愛用者」「購入者」「リピーター」「投稿者」「夢中」「熱中」
  - **OK**: 「モニター」「体験者」「利用者」「使用者」「モニターボイス」

#### 薬機法
- 用法・用量の指定表現はNG
  - **NG**: 「食前に～」「食前習慣」「1回4粒」
  - **OK**: 「食前などの」「食前目安」「1回4粒目安」
- 効果効能の断定的表現は不可

#### 景品表示法（虚偽データ関連、2024年3月～）
- 虚偽の受注速報コンテンツ → **一切禁止**
- 虚偽の「今○○人がこのページを見ています」 → **一切禁止**
- アンケート未集計のまま「○○%が選択」 → **禁止**
- ビヨンドのウィジェットとして提供されていても一切利用不可
- 受注速報はアール社支給パーツのみ利用可

#### 行政固有名詞の使用禁止（2024年1月～）
- 「消費者庁」等の行政固有名詞 → 全案件禁止

### 2-C. 媒体別レギュレーション

#### Meta/Facebook
- お腹の脂肪系文言 + お腹素材の同時使用 → **NG**
- お腹の脂肪系文言のみ → OK（次シーンにお腹素材はOK）
- ビフォーアフター風 → モザイク/カラーフィルター等で対策（色違い×4作成で1/4程度通過）
- 広告却下・垢バン時 → 否認されたものを停止/削除、垢バンはFBページ利用停止/削除
- **広告アカウント名にも効果効能を避けた文言を使用**（レギュレーション違反で停止事例あり）

#### Google
- 医薬品審査フォーム提出必要（ファーマフーズ・明治薬品ともに手順あり）
- 再審査請求フローあり

#### ByteDance/TikTok
- 審査精度が高い
- SPC（一部表現）は他媒体ではOKでもBDではNG → 審査基準統一に2ヶ月程度かかる
- WEB面除外はホワイトリスト申請（アカウント単位）
- 医薬品は記事LP可能に（パングル）

#### LINE
- 審査否認理由の問い合わせフローあり

#### SmartNews
- 独自審査基準あり（`スマニュー審査基準.pdf`）

## 3. チェック対象CRタイプと必要チェック項目

### 3-1. バナー（静止画）
- [ ] NG表現セルフチェック（△ or ×記載）
- [ ] 未チェックフォルダに格納済みか
- [ ] 提出シートに転記済みか
- [ ] 二重チェック完了（最終：案件ディレクター）
- [ ] 完全NG → 削除、微妙 → 提出シートに△/×記載して提出

### 3-2. 動画
- [ ] 動画内テキスト起こしと実際の音声が一致しているか
- [ ] ナレーター文言・注釈も提出シートに記載されているか
- [ ] 体験者コメントに「依頼に基づく内容」表記があるか（ステマ対策）
- [ ] 動画がゴミ箱に入っていないか
- [ ] 修正時：変更部分のテキストを赤色に変更

### 3-3. タイトル（TD）
- [ ] タイトル集シート内で作成されているか
- [ ] NG表現セルフチェック（△ or ×記載）
- [ ] 二重チェック完了（最終：案件ディレクター）

### 3-4. 記事LP
- [ ] 冒頭PR/広告表記あり（11px以上、テキスト表記）
- [ ] 口コミ表現が適切か（モニター/体験者はOK、愛用者/投稿者はNG）
- [ ] オファー情報が遷移先公式LPと一致しているか
- [ ] 受注速報コンテンツがアール社支給パーツか
- [ ] 虚偽の「閲覧人数」「アンケート結果」表示がないか
- [ ] 行政固有名詞がないか
- [ ] 素材使用期限内か
- [ ] 案件固有のNG表現に抵触していないか

### 3-5. 入稿時チェック（全CR共通）
- [ ] 記事URLと遷移先LPの整合性確認
- [ ] AD Ebisの広告ID備考欄に当該LPの内容を明記
- [ ] ダブルチェック者の記名

## 6. システム構成案（エンジニア向け）

### 6-1. データベース設計（必要なマスタ）

```
■ regulation_rules（レギュレーションルール）
- id
- category: enum [法令, クライアント, 媒体, 社内]
- subcategory: string (例: "ステマ規制", "薬機法", "ファーマフーズ共通")
- rule_text: string (ルール内容)
- ng_expressions: json (NG表現リスト)
- ok_expressions: json (OK表現リスト)
- applicable_clients: json (適用クライアント)
- applicable_products: json (適用商材)
- applicable_media: json (適用媒体)
- effective_from: date
- effective_until: date (null = 現在有効)
- severity: enum [絶対NG, 要確認, 推奨]
- source_url: string
- created_at, updated_at

■ ng_cases（過去NG事例）
- id
- client: string
- product: string
- cr_type: enum [バナー, 動画, TD, 記事LP]
- ng_expression: string
- ng_reason: string
- occurred_at: date
- severity: enum [事後NG, 事前NG, ヒヤリハット]
- countermeasure: string

■ check_templates（チェックテンプレート）
- id
- cr_type: enum [バナー, 動画, TD, 記事LP, 入稿]
- client: string
- check_items: json (チェック項目リスト)

■ material_licenses（素材使用期限）
- id
- material_name: string
- person_name: string
- applicable_products: json
- license_until: date
- status: enum [有効, 期限切れ, 使用終了]
```

### 6-2. 機能要件

#### Phase 1: チェックリスト機能
- クライアント × 商材 × CR種別を選択すると、該当するチェック項目一覧が表示される
- 各項目にOK/NG/未確認を記録
- NG項目があれば提出をブロック（警告表示）

#### Phase 2: テキスト自動チェック
- タイトル/記事テキストを入力すると、NG表現データベースと照合
- マッチしたNG表現をハイライト表示
- 代替OK表現を提案

#### Phase 3: 入稿整合性チェック
- 記事URL → 遷移先LP URLの自動取得・比較
- オファー情報の不一致検出
- PR表記の有無・フォントサイズチェック

#### Phase 4: ダッシュボード・通知
- レギュレーション更新履歴の一覧表示
- 素材使用期限のアラート通知
- 案件開始/再開時の必須チェック未完了アラート
- ヒヤリハット共有の集約表示

## 7. 運用フロー（To-Be）

```
【CR制作】
  ↓
【システムにCR情報入力】
  - クライアント/商材/CR種別を選択
  - テキスト・URL等を入力
  ↓
【自動チェック実行】
  - NG表現スキャン
  - チェックリスト表示
  - 入稿整合性確認
  ↓
【手動チェック（二重チェック者）】
  - システム上でOK/NG判定を記録
  - ダブルチェック者の記名
  ↓
【提出可否判定】
  - 全項目OK → 提出許可
  - NG項目あり → ブロック＋理由表示
  ↓
【アール社/CL提出】
  ↓
【チェックバック結果登録】
  - OK → OKフォルダ格納
  - NG → NG事例としてDBに蓄積（自動学習）
```
